# Writing style reference

The highest-priority style guide for blog prose on this site: tone, rhythm,
structure, vocabulary, humor, informality, formatting preferences, and what to
avoid. Apply it before drafting, editing, rewriting, outlining, or reviewing any
post.

## Voice

- Write like a human engineer, not a corporate content machine.
- Keep the writing natural, practical, readable, and human.
- Preserve a personal, slightly opinionated voice — the post should feel like an
  experienced engineer sharing something real and useful.
- Prefer concrete engineering explanations over fluffy inspiration.
- Always write the final post in English, even when the source notes are in
  another language.

## Tone

Encouraged when it fits the material:

- informal phrasing
- light jokes
- mild slang
- opinionated takes
- short, punchy sentences
- practical examples
- honest trade-offs
- "here's what actually happens" explanations

Avoid:

- corporate blog tone
- SEO-stuffed intros
- openers like "In today's fast-paced world..."
- excessive hype and fake enthusiasm
- overexplaining obvious things
- generic AI transitions and filler words: "Moreover", "Furthermore", "Delve",
  "Unlock", "Seamless", "Robust", "Leverage", and similar

## Structure

When writing a full post, prefer this flow (but don't force it if the piece
needs something else):

1. Strong title
2. Short hook / introduction
3. Problem or context
4. Main thesis
5. Explanation with examples
6. Practical takeaways
7. Optional caveats / trade-offs
8. Short conclusion

Use headings, lists, code blocks, blockquotes, and tables when they improve
readability — never just to look busy.

## Technical writing rules

- Be precise. Do not invent facts.
- Mark uncertainty clearly; state assumptions when they are needed.
- Prefer real engineering trade-offs over one-sided claims.
- Include code examples when they make an idea easier to understand.
- Explain jargon the first time it appears.
- Use analogies only when they genuinely clarify the topic.
- Keep examples realistic, not toy-like, unless a toy example is the clearest
  option.

## Quality checklist

Before finishing, confirm the post:

- has a clear point
- starts with a real hook, not filler
- is easy to read aloud
- explains technical ideas plainly
- sounds like a specific human, not a generic AI assistant
- avoids unnecessary buzzwords
- gives the reader something practical to take away
