<Callout title="Context" type="note">
  This came from a real migration in an LLM-powered conversation flow. The first version sent the whole chat history on every turn and hoped the prompt would keep the experience tidy. It worked for the prototype. Then the product became real enough to expose the cost.
</Callout>

The first version looked reasonable.

We had a conversational flow that collected a few structured inputs from a user. Ask a question, get an answer, ask the next question, and eventually send a clean payload to the backend.

Nothing exotic.

So we did the obvious prototype thing: send the whole conversation to the LLM on every turn and tell it what to do.

Something like:

```text
Here is the full chat history.
Ask one question at a time.
Do not repeat yourself.
Collect these fields.
When everything is ready, return the final result.
```

For a demo, this was fine.

Then the flow got longer. The prompt got longer. User answers got messier. The model started asking questions that had already been answered. Sometimes it skipped ahead. Sometimes it treated vague text as complete data. Sometimes a tiny copy change made behavior worse for no obvious reason.

That was the moment the real problem became clear:

> We were asking the model to be both the conversational layer and the source of truth.

Those should not be the same job.

## The prototype became prompt soup

The first architecture was beautifully simple:

```text
User -> LLM with full transcript -> LLM infers state -> response
```

This is attractive because there is almost no plumbing. The model reads everything and decides what comes next.

The problem is hidden inside “decides what comes next.”

On every request, the model had to:

- reread the full transcript
- figure out which fields were already collected
- interpret the latest user message
- decide whether the answer was good enough
- choose the next field
- avoid repeating questions
- keep the tone friendly
- know when the flow was complete

That is a lot of responsibility for a pile of text.

At first, the failures looked unrelated:

- responses got slower
- requests sometimes timed out
- the assistant repeated questions
- required fields were skipped
- validation rules kept leaking into the prompt
- small prompt edits changed behavior in weird ways

But most of them had the same root cause.

The model was reconstructing state from prose every turn.

So every bug became another instruction:

```text
Ask one question at a time.
Do not repeat questions.
Skip fields the user already answered.
Validate the new answer.
If the answer is ambiguous, ask a follow-up.
Do not mark the flow complete too early.
Keep the tone friendly.
Also handle users who answer two fields at once.
Also do not lose previous answers.
Also please be normal.
```

This is how prompts become policy junk drawers.

Every production issue adds a sentence. Every sentence interacts with the previous sentences. After a while, nobody really knows what controls behavior anymore. You just hope the prompt is large and stern enough to keep the model in line.

That is not a great operating model.

## The better split: the model talks, the system knows

The migration was not a deep AI breakthrough. It was mostly a responsibility cleanup.

Instead of asking the model to infer the current state from the entire transcript, we started passing state explicitly.

A simplified request looked like this:

```json
{
  "user_input": "We want this for internal platform users, probably next quarter.",
  "state": {
    "goal": "launch a cleaner onboarding flow",
    "audience": null,
    "timeline": null,
    "success_criteria": null
  }
}
```

The model returned structured output:

```json
{
  "message": "Got it. What would success look like for those internal platform users?",
  "is_ready": false,
  "suggestions": ["Fewer support tickets", "Faster first setup", "Higher activation"],
  "next_field": "success_criteria",
  "state_patch": {
    "audience": "internal platform users",
    "timeline": "next quarter"
  }
}
```

Then the backend validated the patch and persisted the canonical state.

The important boundary was this:

> The model can suggest what changed. The system decides what is true.

That boundary made the flow much easier to reason about.

The backend owned:

- validation
- persistence
- allowed transitions
- completion criteria
- canonical state
- retries and logging

The model owned:

- natural language
- helpful suggestions
- follow-up questions
- handling messy user input
- making the conversation feel less like a tax form

This is a better division of labor.

The model is good at fuzzy language. Deterministic code is better at known facts, allowed transitions, and persistence.

## Why explicit state made the product feel calmer

The first win was speed.

Full-history prompting gets heavier as the conversation grows. This is obvious in hindsight, which is the most annoying kind of obvious.

In this flow, sending explicit state instead of the full transcript reduced the average request from roughly 2,100 tokens to roughly 600. The largest requests went from 5,000+ tokens to around 1,100.

That had a direct effect on latency.

The bigger win was predictability.

When the model inferred state from a transcript, correctness depended on interpretation. If the user answered two things in one sentence, the model might catch both. Or not. If the prompt had a confusing instruction, the model might over-prioritize it. Or ignore it.

With explicit state, the model did not need to rediscover the plot every time.

It already knew:

```json
{
  "goal": "launch a cleaner onboarding flow",
  "audience": "internal platform users",
  "timeline": "next quarter",
  "success_criteria": null
}
```

So the next step became boring:

```text
Ask for success_criteria.
```

Boring is good here.

The user does not care that your prompt is clever. They care that the assistant does not forget what they said thirty seconds ago.

## The architecture became more boring, which was the point

The old flow looked elegant in a diagram:

```text
User -> LLM(full history) -> response
```

It also made the LLM responsible for too much:

```text
LLM = parser + state manager + validator + router + copywriter + database-ish memory
```

That is convenient until something breaks.

The new flow was slightly more mechanical:

```text
User -> Backend loads state -> LLM(state + input) -> structured response -> Backend validates -> persist -> response
```

This adds backend work, but it gives you normal engineering tools back:

- schema validation
- deterministic transitions
- smaller prompts
- better logs
- safer retries
- clearer UI states
- easier debugging

Instead of asking, “Why did the model ask that?” you can inspect the state and transition:

```json
{
  "previous_state": {
    "goal": "launch a cleaner onboarding flow",
    "audience": null,
    "timeline": null,
    "success_criteria": null
  },
  "state_patch": {
    "audience": "internal platform users",
    "timeline": "next quarter"
  },
  "next_field": "success_criteria"
}
```

That is not glamorous. It is useful.

The model still matters. It just stops being the database, reducer, validator, and copywriter at the same time.

## When I would use this pattern

I do not think every LLM interaction needs a state machine.

If the user is asking open-ended questions, brainstorming, summarizing text, or exploring ideas, full conversation history may be completely fine.

The state approach becomes more valuable when the product has structure hiding under the conversation.

I would reach for it when the flow has:

- required fields
- completion criteria
- branching logic
- validation rules
- retries
- progress indicators
- handoff to a backend process
- user-facing consequences if the data is wrong

In other words, if the conversation looks casual but the business logic is strict, do not make the model infer the strict part from prose.

Give it state.

<VisualCard eyebrow="Design heuristic" title="Make ambiguity the model’s job, not truth" tone="accent">
  Let the model handle fuzzy language, tone, suggestions, and recovery. Let deterministic code handle known facts, allowed transitions, readiness, and persistence.
</VisualCard>

The model can say:

> “Almost there. What would success look like?”

The system should know:

```json
{
  "next_field": "success_criteria",
  "is_ready": false
}
```

That combination is the sweet spot.

Friendly on the surface. Boring underneath.

## The trade-off

This approach is not free.

You have to design the state shape. You need validation. You need to decide how strict transitions should be. You need to handle cases where the model returns a bad patch or the user answers out of order.

So yes, full-history prompting is cheaper at the start.

For a prototype, I would still consider starting there. It helps you learn what users say, what fields matter, and where the flow feels awkward.

But once the flow becomes important, the cheap version gets expensive.

The hidden cost of full-history prompting is that complexity does not disappear. It moves into the prompt, where it is harder to test and harder to reason about.

## What this means in practice

The rule I now trust is simple:

> If an LLM flow has required fields, do not ask the model to infer the canonical state from chat history.

Use a state object.

It can be tiny. Three nullable fields and a `next_field` value can be enough. The point is not ceremony. The point is ownership.

A practical implementation can look like this:

```ts
type FlowState = {
  goal: string | null;
  audience: string | null;
  timeline: string | null;
  successCriteria: string | null;
};

type ModelResponse = {
  message: string;
  nextField: keyof FlowState | null;
  isReady: boolean;
  statePatch: Partial<FlowState>;
};
```

Then the backend decides what to accept:

```ts
function applyPatch(state: FlowState, patch: Partial<FlowState>): FlowState {
  const cleanedPatch = removeInvalidValues(patch);

  return {
    ...state,
    ...cleanedPatch,
  };
}
```

The exact implementation depends on the product. You might use a real state machine library, a reducer, a schema validator, or plain code.

The important part is the boundary:

- the model proposes
- the backend validates
- the system persists
- the UI renders from known state

That is much easier to operate than “the answer is somewhere in the transcript, good luck.”

## Final thought

The migration was not really from prompts to state machines.

It was from vibes to boundaries.

Full-history prompting made the model reconstruct the past before it could decide the future. Explicit state let the backend carry the past while the model focused on the next useful sentence.

The result was faster, more predictable, and easier to debug.

The bigger lesson is simple:

> Stop asking the model to remember everything. Tell it what the system already knows.

Your prompt gets shorter. Your flow gets calmer. Your users stop answering the same question twice.
